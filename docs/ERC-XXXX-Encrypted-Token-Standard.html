Abstract

This standard defines a minimal interface for fungible tokens with encrypted balances. Token balances are stored as Fully Homomorphic Encryption (FHE) ciphertexts. Transfers are verified via zero-knowledge proofs without revealing amounts, sender balances, or recipient balances to any on-chain observer.

Unlike ERC-20 where balanceOf returns a plaintext uint256, this standard returns an opaque ciphertext that only the balance owner can decrypt. Arithmetic operations on balances (credit, debit, comparison) are performed homomorphically by the execution environment without any party observing plaintext values.

This standard is backwards-compatible with ERC-20 for public aggregate values (totalSupply, name, symbol, decimals) while introducing encrypted counterparts for all privacy-sensitive operations.

Motivation

ERC-20 tokens expose all balances and transfer amounts publicly. This creates several critical problems:

  1. Portfolio exposure — Anyone can see how much of any token an address holds, enabling targeted attacks, social engineering, and front-running.
  2. Transfer surveillance — Every transfer amount is public, enabling transaction graph analysis, de-anonymization, and MEV extraction.
  3. Governance coercion — In DAO voting with token-weighted governance, visible balances enable vote buying, whale tracking, and political coercion.
  4. Regulatory conflict — Financial privacy regulations (GDPR Article 17, MiCA, HIPAA for health tokens) conflict with permanent public ledger storage.
  5. MEV extraction — Visible pending transactions enable sandwich attacks, front-running, and other MEV strategies that extract value from users.

Why not just use ERC-20 with a privacy layer?

Privacy layers (Tornado Cash, Aztec Connect) are external wrappers. They break composability: a "shielded" token cannot be used in Uniswap, Aave, or Compound without unshielding first — defeating the purpose. This standard makes privacy native to the token interface. A compliant lending protocol can accept encrypted collateral, compute liquidation thresholds homomorphically, and never see the actual collateral amount.

Specification

Definitions

TermDefinition
CiphertextAn FHE-encrypted value. Represented on-chain as bytes. Only the owner of the corresponding decryption key can recover the plaintext.
ProofA zero-knowledge proof (SNARK/STARK) attesting to the validity of an operation without revealing the operands.
Blind TransferA transfer where the amount is encrypted. The proof guarantees: (a) sender has sufficient balance, (b) amount is non-negative, (c) no overflow, (d) balances correctly updated.
Selective DisclosureThe ability for a balance owner to prove a property of their balance (e.g., "≥ 1000") without revealing the exact value.
Re-encryptionTransforming a ciphertext from the contract's public key to the user's personal key, off-chain, so only the user can decrypt.

Core Interface

Solidity Interface IERCXXXX.sol 
// SPDX-License-Identifier: CC0-1.0
pragma solidity >=0.8.0;

/// @title ERC-XXXX Encrypted Token Standard
interface IERCXXXX {

    // ─── Events ───────────────────────────────────

    event BlindTransfer(
        address indexed from,
        address indexed to,
        bytes32 indexed proofHash
    );

    event BlindApproval(
        address indexed owner,
        address indexed spender
    );

    event BlindMint(address indexed to, bytes32 indexed proofHash);
    event BlindBurn(address indexed from, bytes32 indexed proofHash);

    // ─── ERC-20 Compatible (Public Metadata) ──────

    function name() external view returns (string memory);
    function symbol() external view returns (string memory);
    function decimals() external view returns (uint8);
    function totalSupply() external view returns (uint256);

    // ─── Encrypted Balance Queries ─────────────────

    function encryptedBalanceOf(address account)
        external view returns (bytes memory);

    function encryptionScheme()
        external view returns (string memory);

    function publicKey()
        external view returns (bytes memory);

    // ─── Blind Transfers ──────────────────────────

    function blindTransfer(
        address to,
        bytes calldata encryptedAmount,
        bytes calldata proof
    ) external returns (bool);

    function blindTransferFrom(
        address from,
        address to,
        bytes calldata encryptedAmount,
        bytes calldata proof
    ) external returns (bool);

    // ─── Encrypted Approvals ──────────────────────

    function blindApprove(
        address spender,
        bytes calldata encryptedAmount,
        bytes calldata proof
    ) external returns (bool);

    function encryptedAllowance(
        address owner,
        address spender
    ) external view returns (bytes memory);

    // ─── Selective Disclosure (OPTIONAL) ──────────

    function verifyBalancePredicate(
        address account,
        bytes calldata predicate,
        bytes calldata proof
    ) external view returns (bool);

    // ─── Mint / Burn (OPTIONAL) ───────────────────

    function blindMint(
        address to,
        bytes calldata encryptedAmount,
        bytes calldata proof
    ) external;

    function blindBurn(
        bytes calldata encryptedAmount,
        bytes calldata proof
    ) external;
}

Predicate Encoding

enum PredicateType {
    GTE,    // balance >= threshold
    LTE,    // balance <= threshold
    EQ,     // balance == threshold
    NEQ,    // balance != threshold
    RANGE   // lower <= balance <= upper
}

// Encoding:
// predicate = abi.encode(PredicateType, threshold)
// For RANGE: abi.encode(PredicateType.RANGE, lower, upper)

Proof Public Inputs

publicInputs = {
    from:                address,    // sender
    to:                  address,    // recipient
    encryptedAmountHash: bytes32,    // hash of encrypted amount
    oldBalanceHashFrom:  bytes32,    // sender pre-transfer hash
    newBalanceHashFrom:  bytes32,    // sender post-transfer hash
    oldBalanceHashTo:    bytes32,    // recipient pre-transfer hash
    newBalanceHashTo:    bytes32,    // recipient post-transfer hash
    nonce:               uint256     // replay protection
}

Shield / Unshield Bridge

A token MAY implement both ERC-20 and this standard simultaneously, with bridge functions to move between pools:

/// Move tokens from public ERC-20 balance to encrypted balance.
function shield(uint256 amount) external;

/// Move tokens from encrypted balance to public ERC-20 balance.
function unshield(uint256 amount, bytes calldata proof) external;

Rationale

Why encrypted balances instead of commitments?

Commitment schemes (Pedersen, Poseidon) hide values but don't support computation. You can't add two Pedersen commitments and get a valid commitment to the sum without the sender's help. FHE ciphertexts support homomorphic addition and comparison natively, enabling the contract to update balances without any party revealing plaintext values.

Why separate events without amounts?

ERC-20's Transfer(from, to, amount) leaks the amount. BlindTransfer(from, to, proofHash) preserves the minimum information needed for indexing (who transferred to whom) without revealing how much. The proofHash enables off-chain auditors (with appropriate keys) to verify transfer validity.

Why verifyBalancePredicate instead of balanceOf?

In DeFi, you rarely need the exact balance. You need to know "does this user have enough collateral?" — a predicate: balance ≥ threshold. Selective disclosure via predicate proofs gives protocols exactly the information they need, and nothing more.

Composability preserved. A lending protocol implementing this standard can check verifyBalancePredicate(borrower, abi.encode(GTE, minCollateral), proof) to verify collateral sufficiency without ever learning the borrower's actual balance.

Backwards Compatibility

This standard is fully backwards-compatible with ERC-20 for metadata queries. Tokens implementing this standard alongside ERC-20 can participate in both the public and encrypted ecosystems.

Wallets that do not support this standard will display the token name and symbol but will show a balance of 0 (since balanceOf returns the public pool balance, which may be empty if all tokens are shielded).

This standard REQUIRES ERC-165 for interface detection, allowing protocols to query whether a token supports encrypted operations before attempting them.

Security Considerations

Critical: FHE Key Management. The contract's FHE secret key is the most sensitive asset. If compromised, all balances are exposed. Implementations MUST use threshold key management (Shamir SSS with ≥ 5 custodians) or HSMs. The STYX Protocol extends this with the Oblivion ceremony for provable key destruction.

ThreatVectorMitigation
Ciphertext malleability FHE ciphertexts are malleable by design ZK proof binds ciphertext to operation; verify before state update
Proof replay Re-submitting a valid proof Nonce registry; reject previously-used nonces
Side-channel leakage Gas / timing / access patterns Constant-size proofs; uniform circuit compilation
Quantum threat Shor's algorithm on ZK proofs Lattice-based FHE is quantum-safe; use PQ-safe proof system
Key compromise FHE secret key extraction Threshold key mgmt (Shamir SSS ≥ 5); Oblivion ceremony
Share inflation Vault first-depositor attack analog Minimum initial deposit; virtual shares

Metadata leakage. While balances and amounts are encrypted, transaction timing, gas costs, and access patterns remain public. Implementations SHOULD pad all transactions to uniform gas consumption.
